# 文档规范

本项目所有文档统一放在 `docs/` 目录下，**文件名使用中文、内容使用中文**。

## 命名与结构

- 文件名：使用清晰中文命名，避免含糊缩写（如“说明/设计/规范/指南”）。
- 文档入口：统一从 `docs/文档目录.md` 索引进入。
- 一个主题一个文件：避免把多个无关主题混在同一文档里。
- 版本与变更：重要规范/设计变更请在文档顶部增加“变更记录”（可选，但推荐）。

## 书写风格

- 标题层级：最多 3 级标题（`#`/`##`/`###`），便于阅读与检索。
- 术语一致：同一概念全项目统一中文术语（必要时在首次出现处给出英文/缩写）。
- 代码/命令：使用代码块，并保持可复制执行。
- 链接：优先使用相对路径（例如 `docs/架构设计.md`），避免硬编码外部链接。

## 内容要求（建议）

不同类型文档建议包含以下要素：

- **设计文档**：背景 → 目标/非目标 → 方案 → 数据结构/接口 → 风险与取舍 → 迁移计划。
- **规范文档**：范围 → 约束 → 示例 → 常见问题。
- **指南文档**：前置条件 → 步骤 → 验证方式 → 常见报错处理。

## 图与流程

- 允许使用 Mermaid 画图（如时序图/流程图），但需保证纯文本可读。
- 图必须配合文字说明，避免“只有图没有解释”。

## 更新与评审

- 代码行为有变更时，同步更新对应文档（至少更新“项目说明/架构设计/接口约定”之一）。
- 合并前自查：文档是否能独立阅读（不依赖口口相传的背景）。
